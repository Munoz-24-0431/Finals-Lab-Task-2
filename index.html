ER Model to Relational Tables

Project Overview

This project focuses on the transformation of an Entity-Relationship (ER) Model into Relational Database Tables. We'll implement a comprehensive database structure for managing student assignments and submissions, demonstrating the practical application of database design principles. The project showcases the creation of tables with appropriate relationships, primary keys, and foreign keys to ensure data integrity and efficient data management.

Project Objectives

  • Design and implement a normalized database structure
  • Create tables with appropriate primary and foreign keys
  • Establish proper relationships between tables
  • Ensure data integrity through constraints
  • Document the implementation process
  • Demonstrate practical database design principles

Database Schema Design

Table Structures

1. Student Table

  • Primary Key: username (VARCHAR 50)
  • Purpose: Stores unique student identifiers
  • Relationships: One-to-Many with submissions

2. Assignment Table

  • Primary Key: shortname (VARCHAR 50)
  • Fields:
  • due_date (DATE) NOT NULL
  • url (VARCHAR 255) NULL
  • Relationships: One-to-Many with submissions

3. Submission Table

  • Composite Primary Key: (username, shortname, version)
  • Fields:
  • submit_date (DATE) NOT NULL
  • data (TEXT)
  • Foreign Keys:
  • username references student(username)
  • shortname references assignment(shortname)

Step by Step Process

Implementation Steps

1. Create the student table

  • Define username as a VARCHAR(50)
  • Set username as the Primary Key

2. Create the assignment table

  • Define shortname as a VARCHAR(50) and set it as the Primary Key
  • Define due_date as a DATE NOT NULL
  • Define url as a VARCHAR(255), which can be null

3. Create the submission table

  • Define username and shortname both as VARCHAR(50)
  • Define version as an INT
  • Define submit_date as a DATE NOT NULL
  • Define data as TEXT
  • Set a composite primary key of (username, shortname, version)
  • Add foreign keys referencing the student and assignment tables

Table Relationships

1. student table

  • Primary Key: username
  • Relationships:
  • One-to-Many with submission: One student (username) can have many submissions.

2. assignment table

  • Primary Key: shortname
  • Relationships:
  • One-to-Many with submission: One assignment (shortname) can have many submissions.

3. submission table

  • Composite Primary Key: (username, shortname, version)
  • Relationships:
  • Many-to-One with student: Each submission belongs to one student.
  • Many-to-One with assignment: Each submission is for one assignment.
  • This models a Many-to-Many relationship between students and assignments, with version tracking multiple submissions.
